Hybrid Routing & Simulation Logic

This document explains the internal architecture of the "Hybrid" routing engine, which combines static graph algorithms with real-time and historical data.

1. High-Level Architecture

The system is composed of three main services that work together to generate a journey.

graph TD Client[Client / API] --> ConnectionsService subgraph Core Logic ConnectionsService --> GraphService ConnectionsService --> LinkerService end subgraph Data Sources GraphService --> StaticGraph["Static Graph (NetworkX)"] LinkerService --> GTFS["GTFS Database (SQLite)"] LinkerService --> SimulationService SimulationService --> DelayPatterns[Historical Delay Patterns] end

2. The "Hybrid" Decision Logic

The core innovation is in the LinkerService, which decides whether to use live data (if available) or simulate data based on historical patterns.

sequenceDiagram participant CS as ConnectionsService participant LS as LinkerService participant DB as GTFS Database participant Sim as SimulationService CS->>LS: get_trip_details(trip_id, date) LS->>DB: "Get Static Schedule (Stops & Times)" DB-->>LS: Static Details alt Date is Today LS->>LS: "(Future) Query Live API" else Date is Future LS->>Sim: get_delay(train, station, hour) Sim->>Sim: Query 'delay_patterns' table Sim-->>LS: Predicted Delay (e.g. 5 min) LS->>LS: Enrich Trip with Delay Info end LS-->>CS: Final Trip Details

3. Data-Driven Simulation

Instead of random numbers, the simulation is driven by the delay_patterns table, which is populated from gigabytes of historical Deutsche Bahn data.

Schema: (Train Number, Station, Hour) -> Average Delay

Example Query:

SELECT avg_delay FROM delay_patterns 
WHERE train_number='690' 
  AND station_name='Frankfurt (Main) Hbf' 
  AND hour_of_day=14;

4. Routing Flow

The ConnectionsService orchestrates the entire process:

  1. Discovery: Uses GraphService to find a path of stations (e.g., Frankfurt -> Mannheim -> Munich).
  2. Retrieval: For each segment, uses LinkerService to find the best real train.
  3. Validation: Checks transfer times (minimum 5 minutes).
  4. Enrichment: Adds predicted delays to the response.
